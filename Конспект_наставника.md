# Конспект работы с наставником

Сводка договорённостей и разобранных тем (10 июля 2026). Дедлайны — в [README.md](README.md), концепция — в [Концепция_решения_вопросы_и_итог.md](Концепция_решения_вопросы_и_итог.md).

## 1. Формат работы и роли

- Наставник объясняет, проектирует вместе, делает ревью и показывает, **где** ошибка и **почему**; код решения пишет команда.
- **Станислав** — аналитическая модель, схема файла графа, загрузчик (`core/`), скрипт сверки, отчёт.
- **Алексей** (Python слабее) — имитационная модель по **варианту А**: Станислав проектирует каркас и паттерн «узел = процесс с ресурсами и буферами» на примере одного узла, Алексей тиражирует паттерн на остальные типы узлов.
- Решение «тянет ли Алексей симуляцию» принимается по результату его прототипа к 13 июля; запасной вариант Б — Станислав пишет оба ядра, Алексей берёт визуализацию, графики, README, видео, презентацию.

## 2. Ключевое разграничение: аналитика vs симуляция

| | Аналитическая модель | Имитационная модель |
|---|---|---|
| Время | Нет оси времени, только **интенсивности** («штук в час») | Проживает события минута за минутой |
| Вопрос | «Сколько нужно ресурсов и что получится в среднем?» | «Как система ведёт себя во времени?» |
| Видит | Средние потоки, загрузки, штат, парк, площадь | Очереди, блокировки, каскады, пики, отказы, время пребывания товара |
| Роль | **Проектирует** конфигурацию | **Стресс-тестирует** её |

- Правило самопроверки: аналитика отвечает на всё, что формулируется через «в час» и «в среднем», и молчит про всё, где есть «ждёт», «накопилось», «в момент времени».
- «За сколько часов пройдёт партия N» — аналитике доступно (N / пропускная способность). «Сколько времени конкретный товар проводит в центре» — нет: это очереди, зона симуляции.
- Имитационная модель — **не анимация аналитики**, а второй независимый расчётный движок. План «один считает — второй визуализирует» оставлял симуляцию без исполнителя и не закрывал задачу (см. риск №1 в разделе 8 концепции).

## 3. Что такое поток

- **Поток** — скорость счёта через точку графа: число + размерность «сущностей в час» (аналогия: RPS у веб-сервера). **λ** — традиционное имя входного потока; λ = N/t.
- Поток: (1) течёт по рёбрам — у каждого ребра своё значение; (2) пересчитывается узлами в другие единицы (185 палет/ч × 20 = 3 700 КТЯ/ч × 27 ≈ 100 000 товаров/ч — это один поток в трёх единицах, так связаны контрольные числа задачи); (3) делится на развилках (доля nonsort).
- Поток — всегда **пара «число + тип сущности»**. Половина ошибок таких моделей — сложили КТЯ с товарами. Схема файла обязана нести единицы.
- **Баланс узла**: в установившемся режиме втекает = вытекает (в пересчитанных единицах). Один узел — одно уравнение.
- **Циклы** (возврат nonsort, оборот тары) ломают простой проход «от входа к выходу»: уравнения ссылаются друг на друга → система линейных уравнений. Решать либо `numpy.linalg.solve`, либо итеративно до сходимости — выбор обсудить на ревью.

## 4. Три вопроса одной модели

| Дано | Ищется | Имя вопроса |
|---|---|---|
| Конфигурация + поток λ | Загрузки, потоки | Прямой расчёт: «выдержит ли и с каким запасом?» |
| Поток λ (+ топология) | Минимальный штат/техника на узел | **Размерение**: «сколько ресурсов купить?» |
| Конфигурация | Максимальная λ | **Узкое место**: «сколько центр способен пропустить?» |

- Конфигурация = топология + штат + техника (+ параметры преобразования, координаты). **Загрузка — не ручка, а стрелка прибора**: при фиксированных конфигурации и λ она вычисляется однозначно (поток через узел / (штат × производительность)). Ручки — ресурсы и топология.
- Максимальная пропускная способность центра = λ, при которой первый узел упирается в 100%; этот узел — узкое место.
- Цепочка расчёта: потоки → ресурсы (округление вверх) → загрузка → суммы (штат, парк из транспортной работы «поток × плечо», площадь).
- Загрузка ~95% — система без запаса; выбор запаса (например, целевые ≤85%) — проектное решение, обосновывается в отчёте.

## 5. Буферы

- По умолчанию (п. 2.2 концепции) буфер — **свойство ребра**: ёмкость + единица измерения; переполнение блокирует узел-источник.
- **Правило выбора «ребро или узел»**: буфер между ровно двумя узлами — параметр ребра; буферная зона с несколькими поставщиками и потребителями (например, приёмочное хранение) — **узел-хранение** с пустой функцией преобразования (1 палета → 1 палета), ёмкостью, площадью, ресурсами.
- В установившемся режиме **средний поток не зависит от ёмкости буфера** — буфер влияет на переживание неравномерности (зона симуляции). Аналитике буферы нужны для: (1) площади; (2) стартовой рекомендации ёмкости («X минут потока»).

## 6. Параллельная работа: три контракта

1. **Схема файла графа** — интерфейс между моделями; фиксируется до параллельной работы.
2. **Общий мини-граф (3–5 узлов)** — один файл: прототип для Алексея, первый вход для аналитики, точка сверки.
3. **Формат метрик** — обе модели выгружают результат одинаково (например, CSV: `объект, метрика, значение, единица`), чтобы сверка была скриптом.

Архитектура — **одно приложение, два движка**:

```
config/base.yaml → core/ (загрузка + валидация) → analitic/  → таблицы балансов
                                                ↘ simulator/ → метрики во времени
                                                                ↘ validation/ (сверка)
```

- Общее: файл графа, код чтения/валидации (один, не дублировать!), формат метрик.
- Раздельное: сами вычисления. Симуляция **не подглядывает** в ответы аналитики и наоборот — иначе сверка проверяет «согласна ли модель сама с собой» и не значит ничего.
- Git: ветки (`analytic-model`, `sim-core`), merge в `main`, `git pull` перед работой. Изменения в `config/` и общем ядре — только по договорённости голосом.

## 7. Сверка (контрольная точка 19 июля)

- Идея: два независимых способа посчитать одно и то же обязаны сойтись. Аналитика считает мгновенно, симуляция измеряет за 24 модельных часа; результат — таблица «узел/ребро — аналитика — симуляция — расхождение %».
- Пример: вход 10 палет/ч, депалетизация 1→20, 2 оператора по 6 палет/ч. Аналитика: поток 200 КТЯ/ч, загрузка 10/12 ≈ 83%. Симуляция должна намерить ~то же.
- Подводные камни:
  1. **Прогрев**: симуляция стартует пустой — переходный период отбрасывается до усреднения. Равномерное занижение на 5–10% — признак забытого прогрева.
  2. Сверка честна только при загрузках < 100%. При перегрузе симуляция показывает растущую очередь и потолок мощности — это не ошибка, а её работа; перегруз — отдельный демонстрационный сценарий.
  3. Формат выгрузки метрик согласовать заранее (см. контракт 3).

## 8. Задание 1 (Станислав, до 13 июля): схема графа

Два артефакта: (1) документ схемы — поля узла, ребра, ресурса, генератора входного потока, с типами и единицами; (2) пример-файл мини-графа руками — дыры в схеме вылезают сразу.

Чек-лист ловушек:
- **Единицы измерения** везде: ёмкость буфера — штук *чего*; как ребро знает, что по нему течёт.
- **Где живут доли** (nonsort, 80/20 тары) — свойство узла, ребра или сценария? Выбор аргументировать.
- **Схема vs сценарий**: интенсивность входа, профиль Парето — не свойства графа, если один граф гоняется под номиналом/пиком/отказом.
- **`schema_version`** в файле.
- Финальная проверка: на каждое поле — ответ «кто его потребляет» (аналитика/симуляция/оба). Поле без потребителя выкидывается.
- В документ добавить раздел «Метрики для сверки»: 3–4 величины с единицами.

## 9. Задание 2 (Алексей, до 13 июля): прототип движка

Цепочка `источник → буфер (ограниченный) → обработка → буфер → сток` на SimPy, затем то же на Salabim. Критерий готовности — воспроизведена **блокировка**: обработка медленнее источника → буфер заполняется → источник встаёт.

Минимум SimPy: `simpy.Environment` и `env.run(until=...)`; процесс = генератор (`yield env.timeout(...)`); `simpy.Store(env, capacity=N)` — ограниченный буфер, `put` в полный Store сам приостанавливает источник (блокировка бесплатно); `simpy.Resource(env, capacity=N)` — занятость исполнителей. Читать: simpy.readthedocs.io, «Basic Concepts» + пример «Machine Shop». Критерий выбора движка: удобство ограниченных буферов и анимации (у Salabim 2D-анимация встроенная).

## 10. Решённое упражнение на потоки

Вход 100 палет/ч; депал 1→20; вскрытие 1 КТЯ → 27 товаров + 1 пустой КТЯ; nonsort 5%:
депал → вскрытие: **2 000 КТЯ/ч**; вскрытие → сортировка: **54 000 товаров/ч**; вскрытие → упаковка: **2 000 пустых КТЯ/ч**; сортировка → nonsort: **2 700 товаров/ч**. ✔

**Задачка со звёздочкой (открыта):** упаковка пакует по 27 товаров в КТЯ — хватает ли ей 2 000 пустых КТЯ/ч? Посчитать потребность, сравнить; затем учесть 20% брака тары (п. 4 концепции) и посмотреть, что это делает с балансом. Это вход в цикл тары, где проход «слева направо» перестаёт работать.

## 11. Правила проекта и календарные принципы

- **Все файлы — UTF-8, без исключений** (README дважды ломался UTF-16). В VS Code: `files.encoding = utf8`; из PowerShell писать только с `-Encoding utf8`. YAML-конфиг в UTF-16 сломает `yaml.safe_load` у обоих.
- Последние три дня (30 июля – 1 августа) — только отчёт, видео, презентация; код заморожен. Что не готово к 29-му — режется.
- Ловушка скоупа: код честно читает любой граф, но для отчёта руками собирается и вычищается **одна** обоснованная конфигурация. «Любые конфигурации» — свойство архитектуры, а не обещание в демо.
- Отчёт с аналитической моделью обязателен («симуляция без отчёта не принимается») — Станислав на критическом пути.
