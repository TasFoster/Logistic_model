# Имитационная модель (core/simulator)

Дискретно-событийная симуляция сортировочного центра на **SimPy**. Читает тот же
файл графа, что и аналитическая модель, и проживает его во времени: очереди,
ограниченные буферы, блокировки, каскадная деградация, метрики на интервалах.

Статус: **прототип от 13.07** (контрольная точка «выбор движка» по плану в
[README проекта](../../README.md)). Ядро «файл графа → симуляция → метрики»
работает и проверяется тестами.

## Стек

- Python 3.10+, **SimPy 4** (в разрешённом перечне ПО).
- Стандартная библиотека для метрик/CSV. NumPy/pandas не требуются для ядра.

Установка: `pip install -r core/simulator/requirements.txt`

## Запуск

```bash
# демонстрация блокировки (критерий готовности прототипа)
python -m core.simulator.demo_blocking

# прогон модели на мини-графе (100 000 товаров/ч, 1 модельный час)
python -m core.simulator.run

# прогон с замкнутым циклом оборота тары 80/20 (лучше 3+ часа для выхода на режим)
python -m core.simulator.run --graph core/simulator/graph_tare.json --hours 4 --out results_tare/

# параметры
python -m core.simulator.run --graph core/simulator/graph_mini.json \
                             --hours 12 --warmup 600 --seed 42 --out results/

# самопроверка
python -m core.simulator.test_smoke
```

Результаты: `results/metrics.csv` (формат сверки `object, metric, value, unit`)
и `results/series_minute.csv` (производительность узлов по минутам).

## Что моделируется (мини-граф)

Цепочка `Приёмка → Вскрытие КТЯ → Сортировка → Упаковка` плюс побочные потоки:

- **Ограниченные буферы на рёбрах** — переполнение блокирует узел-источник.
- **Узкое место**: Сортировка настроена чуть медленнее входа → её буфер
  переполняется, вскрытие и приёмка блокируются (каскад вверх по цепочке).
- **Поток nonsort**: доля отбраковки на сортировке уходит в отдельный сток.
- **Поток пустой тары**: вскрытие КТЯ порождает пустой КТЯ (побочный выход).
- **Метрики**: по узлам — throughput, загрузка, время блокировки, число воркеров;
  по рёбрам — средняя/макс. очередь и пик заполнения буфера; по системе — вход,
  выходные потоки, время пребывания.

Проверка на прогоне 1 ч: узкое место — Сортировка (≈100% загрузки), вскрытие
блокируется ~31% времени, доля nonsort ≈ 5%, пустой тары ≈ столько же, сколько
вскрыто КТЯ. Все балансы согласованы (см. `test_smoke.py`).

## Архитектура кода

| Файл | Назначение |
|---|---|
| `model.py` | ядро: `Node` (воркеры-процессы), `Rib` (ограниченный буфер), `SortingCenterModel` |
| `metrics.py` | сбор метрик и выгрузка CSV в формате сверки |
| `run.py` | запуск из командной строки + сводка в консоль |
| `demo_blocking.py` | минимальная демонстрация блокировки буфера |
| `test_smoke.py` | дымовые тесты физической согласованности (8 проверок) |
| `graph_mini.json` | мини-граф прототипа (схема как в `config/`, + `service_time_s`) |
| `graph_tare.json` | граф с замкнутым циклом оборота тары 80/20 (сборка, развилка, источник) |

## Типы узлов (`type_node`)

| Тип | Поведение |
|---|---|
| `Input` | точка приёмки: внешний генератор кладёт входную сущность (палеты) |
| `transform` | 1 вход → выходы по `type_output[i]` × `transform_kof[i]` |
| `sort` | сортировка: доля `nonsort_share` уходит в отбраковку |
| `split` | развилка по долям `params.routes` (тара 80% реюз / 20% брак) |
| `pack` | упаковка со сборкой `assembly` (27 товаров + 1 КТЯ → 1 полный КТЯ) |
| `source` | генератор тары (машина новых КТЯ), восполняет буфер по низкому уровню |

## Цикл оборота тары 80/20 (graph_tare.json)

Замкнутый контур тары (Концепция, раздел 4):
- вскрытие даёт пустой КТЯ на каждый обработанный → развилка **80% в реюз / 20% в брак**;
- упаковка **собирает** 27 товаров + 1 пустой КТЯ в 1 полный КТЯ;
- **машина новых КТЯ** восполняет дефицит тары (выработка эмерджентна = спросу минус реюз).

Проверка на прогоне 4 ч: реюз ≈ 80%, брак ≈ 20%, баланс «реюз + новые КТЯ ≈ спрос
упаковки» сходится в пределах 3% (см. `test_smoke.py`). Новых КТЯ выходит меньше, чем
уходит в брак, — потому что ~5% товаров уходят в nonsort и не требуют тары.

## Пробелы схемы (на согласование со Станиславом)

Прототип-потребитель вскрыл в общем контракте графа то, чего не хватает
имитационной модели (по методике «поле без потребителя выкидывается»):

1. **Нет времени обработки узла.** Дискретной модели нужно `service_time_s`
   (секунд на одну сущность одним воркером). Временно добавлено в `graph_mini.json`
   как расширение — предлагаю внести в общую схему.
2. **`effecive_ellements[].ef` — количество или производительность?** Здесь
   трактуется как **число параллельных воркеров**. Нужно зафиксировать в схеме
   единицу измерения (см. чек-лист ловушек в Конспекте, раздел 8).
3. **Единицы `transform_kof`.** Пример `config/example_graph.json` местами
   несогласован (число коэффициентов ≠ числу выходов, `Proguct`/`Product`).
   Ядро терпимо к этому (недостающий коэффициент = 1), но для сверки нужен
   вычищенный граф.
