# Имитационная модель (core/simulator)

Дискретно-событийная симуляция сортировочного центра на **SimPy**. Читает тот же
файл графа, что и аналитическая модель, и проживает его во времени: очереди,
ограниченные буферы, блокировки, каскадная деградация, метрики на интервалах.

Симуляция **ничего не подставляет от себя** — все параметры берутся из файла графа.

## Стек

- Python 3.10+, **SimPy 4** (в разрешённом перечне ПО).
- Стандартная библиотека для метрик/CSV.

Установка: `pip install -r core/simulator/requirements.txt`

## Запуск

```bash
# основной граф: двухстадийная сортировка 20x20, 400 направлений, оборот тары
python -m core.simulator.run --graph core/simulator/graph_2stage.json --hours 3

# сценарии: пик, отказ узла, рост nonsort
python -m core.simulator.scenarios --graph core/simulator/graph_2stage.json --hours 3

# 2D-схема центра -> results/plan.html
python -m core.simulator.viz --graph core/simulator/graph_2stage.json --hours 3

# одностадийный эталонный граф
python -m core.simulator.run --graph core/simulator/graph_contract.json --hours 3

# пример формата из общего контракта (мощности в нём игрушечные — линия стоит)
python -m core.simulator.run --graph config/example_graph.json --hours 2

# демонстрация блокировки буфера
python -m core.simulator.demo_blocking

# самопроверка
python -m core.simulator.test_smoke
```

Результаты: `results/metrics.csv` (формат сверки `object, metric, value, unit`)
и `results/series_minute.csv` (производительность узлов по минутам).

## Как читается граф

| Поле | Смысл в симуляции |
|---|---|
| `input: {тип: N}` | сколько сущностей каждого типа нужно на одно срабатывание (**сборка**: 27 товаров + 1 короб) |
| `output: {тип: N}` | `N >= 1` — детерминированное размножение (1 КТЯ → 27 товаров + 1 короб) |
| `output: {тип: доля}` | `доля < 1` — **вероятностная развилка** (сортировка: 0.95 товар / 0.05 nonsort) |
| `effecive_ellements[].ef` | **производительность** единицы ресурса, шт/ч → **время обработки = 3600 / ef** |
| несколько элементов | несколько параллельных исполнителей; мощность узла = `sum(ef)` |
| `storage` на ребре | **ёмкость буфера**: при переполнении узел-источник блокируется |
| узел без ресурсов | мгновенный приём (`Storage`) — не создаёт ложного узкого места |

**Время в узле выводится из производительности** (`3600/ef`), отдельный слой времён не нужен.
Поле `service_time_s` в узле поддерживается как переопределение — для сценариев
(«узел стал медленнее», отказ оборудования).

## Типы узлов (`type_node`)

| Тип | Поведение |
|---|---|
| `Input` | точка входа: генератор подаёт `type_input` (палеты) с интенсивностью из `input_stream` |
| `transform` | 1 вход → выходы по `output` |
| `sort` | сортировка; доли в `output` дают отбраковку nonsort |
| `split` | развилка по долям (тара: 80% реюз / 20% брак) |
| `pack` | упаковка со сборкой (27 товаров + 1 короб → 1 полный КТЯ) |
| `Storage` | сток (nonsort, брак тары) |
| `source` | машина новых КТЯ: восполняет тару в буфере `params.target_rib` до уровня `params.low` |

## Метрики

По узлам: throughput, **работа / блокировка / голодание / отказ** (в сумме ровно 100%
времени исполнителей). Разделение важно: *блокировка* = выходной буфер полон (узел
заперт снизу), *голодание* = не хватает входа (узел ждёт других), *отказ* = узел
выведен из строя сценарием. Узкое место — узел с максимальной суммой «работа + блокировка».

Агрегация на интервалах **1 мин / 1 ч / 12 ч / 24 ч** (требование критериев) —
файлы `series_1min.csv`, `series_1h.csv`, `series_12h.csv`, `series_24h.csv`.
Прогон 24 модельных часов занимает ~215 с и 6 МБ памяти.

По рёбрам: средняя и максимальная очередь, пик заполнения буфера.
По системе: вход, выходные потоки, время пребывания.

## Что модель нашла в примере формата

`config/example_graph.json` — пример описания формата, мощности в нём игрушечные
(`ef: 24` = 24 шт/ч против 100 000 нужных), поэтому линия закономерно стоит.
Но кроме мощностей вскрылась **структурная проблема, не зависящая от чисел**:

**Излишкам пустой тары некуда деваться.** Вскрытие даёт 27 товаров + 1 короб, но ~5%
товаров уходят в nonsort → упаковка собирает партии реже и потребляет **на ~5% меньше
коробов**, чем производит вскрытие. Короба копятся, буфер забивается, вскрытие встаёт
и душит линию. Проверено экспериментом (на реалистичных мощностях):

| Вариант | Выход | Блокировка вскрытия | Буфер коробов |
|---|---|---|---|
| Без развилки тары (nonsort 5%) | 1151 КТЯ/ч | **44.7%** | **полон** |
| Тот же граф без nonsort (проверка) | 1840 КТЯ/ч | 15.4% | 26/100 |

Затор по таре стоил **~60% пропускной способности**.

## Решение: три новых узла (в `graph_contract.json`)

| Узел | Роль |
|---|---|
| `TareSplit` (`split`) | развилка пустой тары: 80% на повторное использование, 20% в брак |
| `StorageBrak` (`Storage`) | сток брака тары (вывоз повреждённых коробов) |
| `MashinaNovyhKTY` (`source`) | машина новых КТЯ, восполняет дефицит тары |

Результат прогона `graph_contract.json` (3 ч):

```
узел                 throughput    работа   блокир.     голод
Depaletizciya             185 шт/ч     74.0%      0.0%     26.0%
unKTU                    3704 шт/ч     92.3%      0.0%      7.7%
Sorting                100002 шт/ч     90.6%      0.0%      9.4%
Pack                     3517 шт/ч     58.4%      0.0%     41.6%
TareSplit                3704 шт/ч     73.8%      0.0%     26.2%
StorageBrak               774 шт/ч        —         —         —
MashinaNovyhKTY           618 шт/ч        —         —         —
выход KTY_full           3505 шт/ч
```

Сортировка выходит ровно на **целевые 100 000 товаров/ч**, блокировок нет,
баланс тары сходится (реюз + новые ≈ спрос упаковки). Всё закреплено тестами.

## 400 направлений и заполняемость КТЯ

Блок `directions` в графе задаёт 400 направлений отправки со степенным профилем:
показатель подбирается численно так, чтобы **верхние 20% направлений забирали 80%
объёма** (из постановки). Получается настоящий длинный хвост: 1-е направление
~18 600 тов/ч, последнее ~29 тов/ч.

Товар **рождается** с направлением на вскрытии КТЯ и **несёт его** по всей цепочке.
Упаковка работает по правилу **«одна коробка — одно направление»**: копит 27 товаров
ОДНОГО направления в отдельной ячейке-накопителе. Если направление вялое и КТЯ не
набрался за `flush_timeout_s`, он уезжает **недозаполненным** — иначе товар не уедет
никогда.

**Требование к площади:** пик одновременно занятых ячеек — **399 из 400**, то есть
накопитель нужен практически на каждое направление. Это же обоснование двухстадийной
схемы 20×20.

**Ключевой компромисс (прогон 4 ч, чувствительность по таймауту закрытия КТЯ):**

| Таймаут КТЯ | Выпущено КТЯ | Ср. заполненность | Недозаполненных | Новой тары |
|---|---|---|---|---|
| 15 мин | 3943 шт/ч | 23.8/27 (88%) | **23.9%** | **1029 шт/ч** |
| 30 мин | 3552 шт/ч | 26.3/27 (97%) | 8.3% | 638 шт/ч |
| 60 мин | 3462 шт/ч | 27.0/27 (100%) | 0.4% | 547 шт/ч |
| 120 мин | 3460 шт/ч | 27.0/27 (100%) | 0.0% | 545 шт/ч |

Закрывать КТЯ раньше — товар уезжает быстрее, но коробки недогружены: на **14% больше
КТЯ** и почти **вдвое больше новой тары**, а значит больше палет и рейсов. Рабочий
диапазон — 30–60 мин.

## Двухстадийная сортировка 20 × 20 (`graph_2stage.json`)

Первая стадия раскидывает товар по 20 группам направлений, вторая (20 секций) —
по конечным направлениям внутри группы. Ребро в секцию выбирается по группе
направления (поле `dest_group` на ребре).

**Вывод 1. Группировка направлений подряд разрушает схему.** Из-за Парето в первую
группу попадают самые жирные направления:

| Группировка | Выход КТЯ | 1-я стадия заблокирована | Загрузка секций |
|---|---|---|---|
| `balanced` (по объёму) | **3391 шт/ч** | 0% | 65–79% (ровно) |
| `sequential` (подряд) | **1480 шт/ч** | **60%** | 6–99% (перекос) |

Наивная нарезка теряет **56% пропускной способности**.

**Вывод 2. Секции нельзя делать одинаковыми.** Самое жирное направление одно даёт
~18 600 тов/ч — вчетверо больше средней нагрузки секции (4 750), а направление
неделимо. Мощности секций в эталонном графе: от 4 000 до 25 000 шт/ч.

## Сценарии (`scenarios.py`)

Граф — это **конфигурация** центра, сценарий — **условия** работы. Один граф гоняется
под разными условиями:

```bash
python -m core.simulator.scenarios --graph core/simulator/graph_2stage.json --hours 3
```

| Сценарий | Вход тов/ч | Выход КТЯ/ч | Δ к номиналу | Узкое место | Товар в центре |
|---|---|---|---|---|---|
| Номинал | 99 900 | 3 391 | — | unKTU (92%) | 9.6 мин |
| Пик +50% | 149 940 | 3 465 | **+2.2%** | Депалетизация (100%) | **35.3 мин** |
| Пик +100% | 199 980 | 3 463 | +2.1% | Депалетизация (100%) | **49.6 мин** |
| Рост nonsort 15% | 99 900 | 3 027 | −10.7% | unKTU (92%) | 10.8 мин |
| Отказ секции (1 ч) | 99 900 | 2 333 | **−31.2%** | unKTU (97%) | 39.5 мин |
| Отказ вскрытия (30 мин) | 99 900 | 2 841 | −16.2% | Депалетизация (90%) | 27.4 мин |

**Главный вывод: пик не увеличивает выход — он превращается в очередь.** При росте
входа в 1.5–2 раза выход упирается в потолок (~3.5 тыс. КТЯ/ч, +2%), а время
пребывания товара в центре растёт с 9.6 до 35–50 минут. Центр спроектирован почти без
запаса: узкое место (вскрытие КТЯ) уже на номинале загружено на 92%.

**Отказ одной секции 2-й стадии на час стоит 31% выхода** — каскад бьёт по всей линии.

## Визуализация 2D (`viz.py`)

```bash
python -m core.simulator.viz --graph core/simulator/graph_2stage.json --hours 3
# -> results/plan.html
```

Один самодостаточный HTML (без интернета и внешних библиотек — важно для проверки на
изолированном сервере): план центра по координатам `pos`, тепловая карта загрузки
узлов, толщина рёбер по потоку, красная подсветка переполненных буферов и отказов,
панель метрик. Наведение на узел/ребро показывает подробности.

## Возвратный поток nonsort

Товары, не прошедшие автосортировку (форма, хрупкость, нечитаемый штрихкод),
не пропадают в стоке, а идут в отдельную ветку:

`Sort1 → Nonsort → RuchnayaSort (ручной разбор) → ┬→ на упаковку (возврат в поток)`
`                                                 └→ роллкейджи → отгрузка`

- **Производительность ручной сортировки — 250 товаров/ч на человека** (из постановки).
  На поток nonsort ≈ 4 800 тов/ч нужно **24 человека**.
- Половина разобранного **возвращается в исходящий поток** (на упаковку): направление
  уже определено вручную. Половина (крупногабарит) уезжает **в роллкейджах**.

Возврат поднял выход центра с 3 391 до **3 530 КТЯ/ч**: 2 600 товаров/ч, которые
раньше терялись, теперь доезжают до отгрузки.

**Почему возврат идёт на упаковку, а не во вход сортировщика.** Обе топологии
работоспособны (блокировок нет ни в одной), но повторный прогон через автомат зря
съедает его мощность: возврат на упаковку даёт **3 343 КТЯ/ч** против **3 228 КТЯ/ч**
при возврате во вход сортировки.

## Мобильные ресурсы: погрузчики (`resource_pools`)

По концепции (раздел 2.3) погрузчики — **не узлы графа**, а пул, обслуживающий
транспортные рёбра. Ребро с полем `pool` копит партию (`batch`), запрашивает свободную
единицу из пула, та едет туда-обратно и выгружает груз.

```json
"resource_pools": { "forklift": {"count": 6, "speed_mps": 2.0, "handling_s": 60} },
"ribs": { "rib_rollcage": {"..." : "...", "pool": "forklift", "batch": 100} }
```

Погрузчиками обслуживаются вывоз брака тары и роллкейджи с крупногабаритом.
Метрики парка: загрузка, рейсов/ч, перевезено/ч, ожидание свободной единицы.
На эталонной конфигурации: **6 погрузчиков, загрузка 33%, 52 рейса/ч**.

**Важно: партию на рейс надо размерять.** Если задать её слишком мелкой (везти по 8
товаров вместо 100), погрузчики становятся узким местом и **каскадом обрушивают весь
поток** — выход падает вшестеро. Закреплено тестом `test_forklift_shortage_chokes_line`.
